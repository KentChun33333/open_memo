The core of this evolution relies on a **Nanobot from HKUD**, a highly capable personal AI agent running locally, and orchestrated via **Telegram**.

This post explores the full component-level architecture of how OpenMemo functions as an automated cognitive system, allowing me to build, deploy, and interact with my knowledge base effortlessly.

### 1. Frontend: React & MDX
The OpenMemo user interface is built entirely with **React**. For rendering the interactive diagrams and rich text, I rely heavily on **React Flow** and support for both standard Markdown (`.md`) and extended **MDX** (`.mdx`). This allows me to seamlessly embed live React components (like the architectural interactive flowchart above) directly inside my blog posts.

### 2. Backend Engine: Local LLMs & Searxng
The underlying backend intelligence is powered by my customized version of Nanobot from HKUD, safely running on local resources. By harnessing local LLMs through Ollama, I maintain absolute command over operational privacy and computing costs. For web discovery, the agent integrates with **Searxng**, an internet metasearch engine chosen specifically because it is completely free to self-host and query. 

### 3. Custom Agent Skills
The true power of this system is how the local loop interacts with my globally accessible codebase. I built **customized agent skills** into Nanobot that allow it to operate autonomously. These tools enable the agent to do everything from creating animations to performing full-scale **DevOps** (reading, modifying, and deploying code without manual intervention).

### 4. Cloud Deployment on GCP Cloud Run
When I ask Nanobot to create a new blog, it gathers context, writes the `.mdx` content, and automatically pushes directly to the remote GitHub repository. This push event seamlessly triggers **GCP Cloud Run** to redeploy the live webapp. I chose GCP Cloud Run because the first 1 million requests per month are completely free, making it incredibly cost-effective to host this autonomous web application.
